# Teal — Architecture

This document is the canonical map of the Teal codebase. **It must be updated in the same commit as any structural change** (new package, new dependency direction, renamed domain term, new module boundary). If you are reading this and it looks out of date, fix it before continuing.

For product-level requirements see [`docs/prompts/01_project.md`](docs/prompts/01_project.md). This file describes _how_ the system is built; the spec describes _what_ it does.

---

## System overview

Teal is a single Go binary plus a SvelteKit frontend, deployed as one docker-compose stack. It runs alongside Docker Engine on a Linux host and orchestrates user apps as docker-compose projects, flipping traffic between two parallel stacks (`<app>-blue` and `<app>-green`) through Traefik to achieve zero-downtime deploys.

```
                          ┌─────────────────────────────────────┐
                          │            Operator (UI / API)       │
                          └────────────────┬─────────────────────┘
                                           │ HTTPS
                          ┌────────────────▼─────────────────────┐
                          │             Teal binary              │
                          │                                      │
                          │   api ──► auth ──► docker ──► (Docker Engine)
                          │    │       │        │                │
                          │    └──► store (SQLite)               │
                          │           ▲                          │
                          │   config / logging (cross-cutting)   │
                          └────────────────┬─────────────────────┘
                                           │ writes dynamic config
                          ┌────────────────▼─────────────────────┐
                          │              Traefik                 │
                          └────────────────┬─────────────────────┘
                                           │ routes
            ┌──────────────────────────────┼──────────────────────────────┐
            │                              │                              │
   ┌────────▼────────┐           ┌────────▼────────┐           ┌────────▼────────┐
   │  myapp-blue     │           │  myapp-green    │   ...     │   otherapp-…    │
   │ (compose stack) │           │ (compose stack) │           │ (compose stack) │
   └─────────────────┘           └─────────────────┘           └─────────────────┘
```

Phase 1 implemented the bottom half of the binary: config, logging, store, domain types, read-only Docker wrapper, HTTP skeleton.

Phase 2 added: real session + API-key auth (cookies, CSRF, bcrypt, login rate-limiting, role gating), full user CRUD, audit-log writes, the SvelteKit frontend (SPA, embedded into the binary at release time), and a session-cleanup goroutine.

Phase 3 added: the deployment engine. Compose YAML transformation, Traefik dynamic-config writer, blue-green orchestrator (state machine with in-memory Phase events), per-app deploy lock, three-mode health checker (docker healthcheck, HTTP probe, running-only fallback). New API surface: full Apps CRUD + deploy + rollback + per-deployment phase.

Phase 4 added: Git/GitHub integration. SSH deploy keys (Teal generates ed25519 keypair per app; the public half is shown to paste into GitHub), PAT (token stored encrypted), or public clone. Fresh shallow clone per deploy with commit SHA resolution; compose read from `<repo>/<git_compose_path>` when configured. GitHub webhook receiver at `/api/v1/webhooks/github/{slug}` with constant-time HMAC-SHA256 validation; branch matching against the app's effective branch; auto-deploy toggle. New crypto primitive `Codec` (HKDF-derived per-purpose keys, shared by engine + API). New frontend Settings tab with git source config, SSH key copy-to-GitHub, and webhook URL/secret surfacing.

Phase 5 added: env-var hydration, platform settings, HTTPS/ACME, and volume management. Per-app and shared env vars are encrypted at rest (`Codec` purposes `envvar.app` and `envvar.shared`, AAD `app:<id>:envvar:<KEY>` and `shared:envvar:<KEY>`); the engine writes a sorted KEY=VALUE `deploy.env` file on every deploy and stamps a SHA-256 hash onto the deployment row. Shared env vars are global but **not** universally injected — each App opts in via an explicit allow-list (`app_shared_env_vars`). New `platform_settings` KV table (admin-editable) stores ACME email, ACME staging flag, and the HTTPS-redirect toggle. The Traefik static config is now generated by Teal at startup and on every settings change (`internal/traefik/static.go`); the per-app dynconf emits HTTPS routers + an optional redirect middleware when ACME is configured. Volume management: per-app filtered list (volumes whose name matches `<slug>-blue_` or `<slug>-green_`) and admin-only typed-name-confirmed delete. New frontend tabs (Env, Volumes) on each app and three new admin pages (`/volumes`, `/settings/platform`, `/settings/shared-env`).

Phase 6 added: real-time streaming + metrics scraping. Three new packages co-operate around a containerwatcher that polls Docker and fans start/stop events to subscribers: `internal/metrics` (scrapes per-container CPU/mem/IO, persists to `metrics_samples`, publishes per sample), `internal/logbuffer` (tails each container's stdout/stderr via `docker logs --follow`, writes one NDJSON file per container with 6-hour retention, publishes per line), and `internal/realtime` (tiny pub/sub hub with per-subscriber buffered channels, slow-client drops). The deploy engine gained a `Publisher` and now emits Phase events + every line written to `deploy.log` on the realtime topic `deploy.<id>`. The API exposes one multiplexed WebSocket at `/api/v1/ws` (built on `coder/websocket`) plus historical-replay HTTP endpoints (`/apps/{slug}/deployments/{id}/log`, `/apps/{slug}/containers`, `/apps/{slug}/metrics`, `/containers/{id}/logs`). Frontend gains a single shared WebSocket client (auto-reconnect, exponential backoff, subscription replay), a `LogStream` component reused for both deploy log + container log views, a hand-rolled `Sparkline` SVG component, a Logs tab on each App, live metrics on the Overview tab, and a deploy-log modal on the Deployments tab.

Phase 7 (current) — polish + installer. New `internal/notify` package dispatches outbound HMAC-signed webhooks (3 attempts, 1s/4s/16s backoff) and SMTP failure emails on every terminal deploy; the engine fires it through a small adapter so the package layering stays clean. New `notifications` table backs the in-app feed; the bell in the topbar subscribes to `notifications.broadcast` (and `notifications.<user-id>`) over WS for live unread counts. Per-app **resource limits** (`cpu_limit`, `memory_limit`) are injected into every service's `deploy.resources.limits` by the compose transform — applied to all services by design so a sidecar can't escape the budget. **Metrics downsampling**: a separate goroutine folds raw samples older than 6h into 1-minute buckets in `metrics_samples_1m`, drops the raw rows, and prunes 1m buckets older than 24h; the metrics endpoint accepts `?resolution=raw|1m`. **Self-update** is a marker-file-and-exit: admin POSTs `/api/v1/platform/self-update?confirm=update-platform`, the binary writes `<workdir>/platform/restart.requested`, exits 0, and the operator's supervisor (systemd / docker `restart: unless-stopped`) brings the new image up. **Dashboard summary widget** shows total apps, running containers, workdir disk usage, and a recent-failures feed. **Installer**: `install.sh` at repo root — two questions only (admin email, base domain), everything else defaulted; generates `PLATFORM_SECRET`, writes `/etc/teal/.env` (0600) + `/etc/teal/docker-compose.yml`, ensures the `platform_proxy` network and a baseline Traefik static config, runs `docker compose up -d`, prints the URL + bootstrap step. `uninstall.sh` companion + `deploy/docker-compose.platform.yml` (the prod compose the installer materialises).

## Module map

The Go module path is `github.com/sariakos/teal/backend`. All non-`main` code lives under `internal/` so it cannot be imported by anything outside this repo.

| Package              | Responsibility                                                                                                | Depends on                                                |
| -------------------- | ------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------- |
| `cmd/teal`           | Process entrypoint. Wires every subsystem in dependency order and handles graceful shutdown. **No logic here.** | every `internal/*`                                       |
| `internal/config`    | Loads typed runtime configuration from environment variables with documented defaults and validation.          | stdlib only                                               |
| `internal/logging`   | Structured slog logger setup. Exposes a request-id context helper used by the API middleware.                  | stdlib only                                               |
| `internal/crypto`    | Symmetric primitives: HKDF key derivation + AES-256-GCM. Used by `internal/auth` (TOTP, planned Phase 7) and `internal/store` (EnvVar values, planned Phase 5). | `golang.org/x/crypto/hkdf`, stdlib |
| `internal/domain`    | Pure Go types representing the core nouns: `App`, `Deployment`, `User`, `EnvVar`, `AuditLog`, `Session`, `APIKey`. No DB, no HTTP. | stdlib only                            |
| `internal/store`     | SQLite connection, embedded migration runner, one repository per aggregate (Apps, Deployments, Users, EnvVars, AppSharedEnvVars, PlatformSettings, Metrics, Notifications, AuditLogs, Sessions, APIKeys). Translates unique-constraint errors into `ErrConflict`. Hides SQL from callers. MetricsRepo also owns the raw→1m downsample (transactional aggregate + drop). | `internal/domain`, `modernc.org/sqlite` |
| `internal/docker`    | Thin wrapper over the Docker Engine SDK. Reads (list containers/networks/volumes, container inspect, container stats one-shot, multiplexed log stream demuxed via `stdcopy`) plus a small set of mutations as features need them: `NetworkCreateIfMissing` (Phase 3 — platform_proxy), `VolumeRemove` (Phase 5 — admin volume cleanup; surfaces `ErrVolumeInUse`), `StreamContainerLogs` + `ContainerStats` (Phase 6 — feeds the logbuffer + metrics scraper). | Docker SDK, `docker/pkg/stdcopy`                          |
| `internal/auth`      | Subject + context, password hashing (bcrypt), session manager (cookies, sliding expiry), API-key manager (SHA-256 hash, base32 token), CSRF middleware, login rate limiter, bootstrap detection, the unified Authenticator middleware. | `internal/domain`, `internal/store`, `golang.org/x/crypto/bcrypt` |
| `internal/compose`   | Pure YAML in/out: parse, lint for blue-green-incompatible patterns, validate via `docker compose -f - config` shell-out, transform (inject platform_proxy + Teal labels, identify primary service). Pure functions; no Docker, no engine state. | `gopkg.in/yaml.v3` |
| `internal/git`       | Shallow clone + SSH keypair generation. `GenerateSSHKeyPair` produces ed25519 PEM + OpenSSH public line; `Shallow` shells out to `git clone` with public/PAT/SSH auth. `PublicKeyFromPrivatePEM` is used by the API to surface the public half from the stored private. | `golang.org/x/crypto/ssh` |
| `internal/github`    | GitHub webhook protocol: HMAC-SHA256 signature verification (constant time) + push-event JSON parser. No HTTP; the API wires it to a chi route. | stdlib only |
| `internal/traefik`   | Owns the platform_proxy bridge network (created at boot), generates the Traefik **static** config from platform settings (`static.go`/`ApplyStaticFromSettings`), and writes per-app **dynamic** config YAML files atomically (`dynconf.go`). Dynconf emits HTTPS routers + a redirect middleware when `RouterSpec.TLSEnabled`/`HTTPSRedirect` are set. The deploy engine calls into the dynamic side during the colour flip; cmd/teal + the API settings handler call the static-config writer. | `internal/docker`, `internal/domain`, `gopkg.in/yaml.v3` |
| `internal/deploy`    | Blue-green deployment engine: lock (DB-backed), workdir (compose+env+log per deploy with retention pruning), runner (shells out to `docker compose`), healthcheck (3 modes), engine (state machine emitting Phase events), env-file hydration (`envfile.go`: collects per-app vars + opted-in shared vars, decrypts each with the shared `Codec`, writes a sorted `KEY=VALUE` file, computes a SHA-256 hash recorded on the deployment row). When `app.GitURL` is set, clones the repo into the workdir before compose transformation and reads compose from `<checkout>/<git_compose_path>`. Reads platform settings to decide whether to emit TLS routers in the per-app dynconf. | `internal/compose`, `internal/traefik`, `internal/docker`, `internal/git`, `internal/crypto`, `internal/store`, `internal/domain` |
| `internal/containerwatcher` | Polls Docker every 2s (default), filters platform-managed containers (Compose project label matches `<slug>-blue` / `-green`), fans `Started`/`Stopped` callbacks to subscribers. The metrics scraper and logbuffer registry both subscribe; this is the single discovery primitive. | `internal/docker`, `internal/domain` |
| `internal/realtime`  | Tiny pub/sub hub keyed by topic. Subscribers get a buffered channel (default 64); slow consumers have events dropped (logged + surfaced via Subscription.Drops()). One Hub per process; multiplexed over a single WebSocket per browser tab. | stdlib only |
| `internal/logbuffer` | Per-container NDJSON log files at `<workdir>/container-logs/<id>.ndjson`. Tailer goroutines call `docker.StreamContainerLogs`, append each line to the file AND publish on topic `containerlogs.<id>`. A prune goroutine trims lines older than retention every 5min and deletes files for containers gone past `IdleDelete`. v1 retention: 6 hours (shared with metrics). | `internal/containerwatcher`, `internal/docker` |
| `internal/metrics`   | Scraper goroutine that polls `docker stats` per known container at the configured interval (default 15s, floored at 5s), persists batched samples via `MetricsRepo`, and publishes each sample on topic `metrics.<id>`. Sibling `Downsampler` goroutine folds raw rows older than `RawRetention` (default 6h) into 1-minute averages in `metrics_samples_1m`, then prunes 1m rows older than `TotalRetention` (default 24h). | `internal/containerwatcher`, `internal/docker`, `internal/store`, `internal/domain` |
| `internal/notify`    | Outbound notification dispatcher. `Notifier` interface called by the deploy engine on every terminal status: writes an in-app `notifications` row + publishes on `notifications.broadcast`, POSTs an HMAC-signed JSON webhook to the per-app URL (3 attempts, 1s/4s/16s backoff, refuses to send unsigned), and on **failure only** sends an SMTP email when platform `smtp.*` settings are configured. | `internal/store`, `internal/crypto`, `internal/domain` |
| `internal/api`       | HTTP server: chi router, standard middleware stack, JSON helpers, resource handlers (`auth`, `users`, `apikeys`, `apps`, `deployments`, `audit`, `docker`, `envvars`, `envvars_shared`, `settings`, `webhooks`, `logs`, `metrics`, `ws`, `frontend`). The `/ws` endpoint multiplexes realtime topics (built on `coder/websocket`) and is the only place using a non-standard upgrade. Optional embedded SPA via build tag `embed_frontend`. **Only place that touches `net/http`.** | every other `internal/*`, `coder/websocket` |
| `frontend/`          | SvelteKit 2 + Svelte 5 + Tailwind v4 + shadcn-svelte SPA. Built as static files (`adapter-static`) and embedded into the Go binary at release time via `make build-release`. | npm                                       |

### Dependency direction (strict)

```
cmd/teal
   │
   ▼
internal/api ──► internal/auth ──► internal/store ──► internal/domain
   │       │                          ▲
   │       ├──► internal/github         (HMAC + push payload)
   │       ├──► internal/git            (keygen via apps handler)
   │       ├──► internal/crypto (Codec; shared with deploy)
   │       └──► internal/deploy ──────┤
   │                  │               │
   │                  ├──► internal/compose
   │                  ├──► internal/traefik ──► internal/docker
   │                  ├──► internal/git    (clone)
   │                  ├──► internal/crypto (Codec open)
   │                  └──► internal/docker
   │
   ├──► internal/docker
   ├──► internal/config ──────────────┘  (config is read by store + api + docker + auth + deploy)
   ├──► internal/logging
   └──► internal/traefik (cmd/teal calls EnsurePlatformNetwork at boot)
```

Rules:

- **Higher-level modules depend on lower-level ones, never the reverse.** `internal/store` may not import `internal/api`; `internal/domain` may not import anything from this repo.
- **No `utils` or `helpers` package.** If you find yourself wanting one, the function belongs in the package whose types it operates on.
- **`internal/api` is the only package allowed to import `net/http`.** All HTTP concerns are funnelled through it.
- **`internal/store` is the only package allowed to write SQL.** Other packages receive domain types, not rows.
- **`internal/docker` is the only package allowed to import the Docker SDK.** Other packages receive plain structs.

These rules exist so that swapping any one subsystem (e.g. SQLite → Postgres, chi → stdlib, Docker SDK → Podman) touches exactly one package.

## Data flow at a glance

A read API call in Phase 1:

```
HTTP request
  → chi router (internal/api/router.go)
    → middleware stack: recover → request-id → logging → JSON content-type → auth
      → resource handler (e.g. internal/api/apps.go)
        → repository (internal/store/apps.go)
          → SQLite
        ← domain.App slice
      ← JSON response
```

A future deploy (Phase 3) will follow the same shape but spawn a long-running deployment goroutine that writes status updates back through the store and broadcasts via WebSocket — see Phase 3 docs when it lands.

## Glossary

These terms are used **identically** in the database schema, Go types, JSON API fields, UI labels, and prose. Do not introduce synonyms.

| Term         | Definition                                                                                                                                   |
| ------------ | -------------------------------------------------------------------------------------------------------------------------------------------- |
| **App**      | One docker-compose project managed by Teal. Has a unique slug, a Git source, env vars, a domain, and a deployment history.                   |
| **Deployment** | A single attempt to bring an App to a target version. Has a `color` (`blue`/`green`), a status, a triggered-by user, a commit SHA, logs.    |
| **Stack**    | The set of running containers belonging to one Deployment. Identified by Compose project name `<app-slug>-<color>`. Not a separate DB row.   |
| **Color**    | `blue` or `green`. Marks which of the two parallel Stacks an App is currently running. The active color is whichever Traefik points at.       |
| **Active color** | The color currently receiving traffic. The opposite color is the deploy target for the next Deployment.                                  |
| **EnvVar**   | A single key/value pair belonging to an App. Encrypted at rest.                                                                              |
| **AuditLog** | An immutable record of a state-changing action: actor, IP, action, target, timestamp.                                                        |
| **Session**  | One active browser-based authentication for a User. Opaque ID in an HttpOnly cookie; CSRF token bound to it; sliding expiry.                |
| **APIKey**   | One programmatic credential for a User. Format `tk_<base32>`. Only the SHA-256 of the raw key is stored; revocation is a timestamp.          |
| **CSRF token** | The synchronizer token bound to a Session. Cookie carries it (non-HttpOnly so the SPA can read), the SPA echoes it via `X-Csrf-Token` on unsafe methods. Bearer-authed requests are exempt. |
| **platform_proxy** | The Docker bridge network Teal owns. Every routed App's primary service is attached to it; Traefik runs on it and proxies to containers by IP. Created idempotently at startup. |
| **Working directory** | Per-deployment dir at `<workdir-root>/deploys/<slug>/<deployment-id>/` containing the transformed compose, the env file, and the captured deploy log. Retained per-app (default last 10) so rollbacks can re-run any kept deployment. |
| **Dynamic config** | Per-app YAML file under `<workdir-root>/traefik/dynamic/<slug>.yml` declaring the router and service Traefik routes for this App. Atomically rewritten by the engine during the colour flip. |
| **Phase** | Fine-grained, in-memory progress state for an in-flight deployment: `pending → pulling/building → starting → healthcheck → flipping_traffic → draining → tearing_down → succeeded/failed`. NOT persisted (the deployment row's coarse Status is). Phase 6 will broadcast these via WebSocket; Phase 3 polls. |
| **Deploy key** | The SSH keypair generated per app when `git_auth_kind == "ssh"`. The ed25519 private key is stored encrypted (AEAD with app-bound AAD); the public half is shown once at creation and derived on demand from the stored private thereafter. |
| **Webhook secret** | The HMAC-SHA256 secret GitHub uses to sign push payloads. Generated server-side the first time git is configured (and on explicit rotation). Stored encrypted; shown raw only once. |
| **Trigger kind** | Why a Deployment exists: `manual` (UI/API call), `webhook` (GitHub push), `rollback` (re-run of a prior successful deployment). Surfaced in the Deployments list. |
| **Env var (app)** | A per-app KEY/VALUE pair encrypted at rest (`Codec` purpose `envvar.app`, AAD `app:<id>:envvar:<KEY>`). Injected into every deploy via an env-file the engine writes per-deployment. |
| **Shared env var** | A platform-wide KEY/VALUE pair (`Codec` purpose `envvar.shared`, AAD `shared:envvar:<KEY>`). Available to apps but never injected automatically — each App opts in by listing keys in `app_shared_env_vars`. Per-app vars shadow shared ones with the same key. |
| **Env-var-set hash** | SHA-256 over the sorted `KEY=VALUE\n` body the engine writes for a deploy. Stored on the Deployment row so two successive deploys with identical env produce identical hashes regardless of insertion order. |
| **Platform setting** | A row in the `platform_settings` KV table edited by admins. v1 keys: `acme.email`, `acme.staging`, `https.redirect_enabled`. ACME keys feed the Traefik static config (Traefik must be restarted to pick them up); the redirect toggle feeds the per-app dynconf (picked up on the next deploy). |
| **Static config (Traefik)** | The boot-time configuration file Teal writes to `<workdir-root>/traefik/traefik.yml`. Declares entrypoints (`web`, `websecure`), the file provider directory, and — when `acme.email` is set — the `letsencrypt` certresolver with HTTP-01 challenge. Traefik reads it once at boot; admins restart Traefik after settings changes. |
| **HTTPS redirect** | Per-router middleware emitted on the HTTP entrypoint when `https.redirect_enabled=true`. Issues a 308 to the same host on `https://`. Toggling it does not require a Traefik restart — the next per-app deploy re-renders the dynconf. |
| **Hub** | Process-wide pub/sub registry (`internal/realtime.Hub`). Holds the subscriber set, fans events out by topic, drops on slow consumer. One per process; multiplexed over `/ws`. |
| **Topic** | String-keyed channel on the hub. Conventions: `deploy.<deployment-id>` (Phase + log lines), `containerlogs.<container-id>`, `metrics.<container-id>`. `_meta` is reserved for server-initiated control frames (drop notices). |
| **Tailer** | One goroutine per running container that reads `docker logs --follow`, demuxes stdout/stderr, appends to the on-disk buffer, and publishes on the hub. Lifecycle managed by the logbuffer Registry, which subscribes to the container watcher. |
| **Sample** | One row in `metrics_samples`: container ID + slug + color + timestamp + CPU%/mem/IO snapshot. Captured by the metrics scraper at the configured interval. Aggregated to `metrics_samples_1m` after `RawRetention`. |
| **Notification** | One in-app feed entry (`notifications` table). Distinct from the audit log: read-state per user, prunable, scoped (per-user OR broadcast-to-admins). The bell renders these. |
| **Outbound webhook** | Per-app URL Teal POSTs to on every terminal deploy. Body is JSON `{event, ts, app, deployment}`; signature in `X-Teal-Signature: sha256=<hex>` keyed by the per-app secret (Codec purpose `webhook.outbound`, AAD `app:<id>:notify`). |
| **Resource limit** | App-wide CPU + memory budget. Injected into `deploy.resources.limits` of every service in the compose transform. Empty values disable. |
| **Self-update marker** | File at `<workdir>/platform/restart.requested` written by `POST /platform/self-update?confirm=update-platform`. The binary exits 0; the supervisor (systemd / docker `restart: unless-stopped`) brings the new image up. |

## Where to find things

- **To add an HTTP endpoint**: `backend/internal/api/`. Add a new file named after the resource if one doesn't exist, register the route in `router.go`.
- **To add a domain type**: `backend/internal/domain/`, then a repository in `backend/internal/store/`, then a migration in `backend/internal/store/migrations/`.
- **To add a database column**: write a new migration file `NNNN_description.sql` in `backend/internal/store/migrations/`. Never edit a migration that has already been applied to any environment.
- **To add a Docker operation**: `backend/internal/docker/`. Keep callers free of the Docker SDK import.
- **To add a realtime topic**: extend whichever package owns the data (engine, logbuffer, metrics) to call `hub.Publish("<topic>.<id>", payload)`. Document the topic name in `internal/realtime/hub.go` so other subsystems can find it.
- **To add a metric**: extend `domain.MetricSample` and the migration; the scraper just reads what `docker.ContainerStats` returns. Keep the column count low — every column is paid per scrape per container.
- **To change runtime configuration**: `backend/internal/config/config.go`. Document the env var name, default, and effect in the struct comment.
- **To add a cross-cutting middleware**: `backend/internal/api/middleware/`.
- **To add an authenticated state-change**: handler should call `recordAudit(...)` after the mutation succeeds. Lossy by design (audit failure does not roll back the action).
- **To embed the frontend in the binary**: `make build-release` (requires Node + npm). Default `make build` does not embed.
- **To add a deployment-engine step**: edit `internal/deploy/engine.go`'s `run` method. Each step emits a `Phase` event before doing work; failures call `e.fail(...)` which tears down the new stack and leaves the old one untouched.
- **To change Traefik routing format**: `internal/traefik/dynconf.go`. Tests cover write-then-parse so a schema change shows up immediately.
- **To bring up Traefik for local dev**: `make -C backend dev-up` (compose stack at `deploy/docker-compose.dev.yml`). Entry point on :80, dashboard on :8080.
- **To add a new per-app secret** (similar to git credential or webhook secret): use `crypto.Codec` with a fresh purpose string and app-bound AAD of the form `"app:<id>"`. Never reuse a purpose for semantically different data — HKDF keys depend on it and a collision is a cryptographic wormhole.
- **To add a new webhook source**: mirror `internal/github` (new package per provider, small surface: HMAC verify + payload parse) and add a route under `/api/v1/webhooks/<provider>/{slug}`.

## Key architectural decisions

These are the load-bearing choices. Each has a one-line rationale. ADRs for non-obvious decisions live in `docs/decisions/` once they accumulate.

- **Go for the backend**: single binary, low memory, first-class Docker SDK. Distribution-friendly.
- **`modernc.org/sqlite` (pure-Go SQLite)** over `mattn/go-sqlite3`: no cgo, trivial cross-compilation, static binary.
- **SQLite** as the only datastore: the spec forbids external dependencies (no Redis, no Postgres). One host, modest write rate — SQLite is sufficient for years.
- **chi router**: middleware composition and sub-routers without dragging in a framework. Stays close to `net/http` semantics.
- **Hand-rolled migration runner** (~80 LOC): the spec asks for a "simple embedded migration runner". Avoids another dependency.
- **Blue-green via two Compose projects** (not a single Compose with replicas): lets us rebuild the target stack independently and tear it down cleanly on failure without touching the live one.
- **Traefik with file-based dynamic config** (not Docker labels alone): the platform owns the routing decision and writes config atomically. Labels would couple routing to container lifecycle, which we explicitly want to decouple for the colour flip.
- **SvelteKit frontend, SPA mode (`adapter-static`)**: the spec wants a single binary distribution; SSR would force a Node runtime alongside Go. Loses SSR's first-paint advantage; an admin UI doesn't need it.
- **`embed.FS` behind a build tag (`embed_frontend`)**: dev `make build` does not require Node; release `make build-release` runs `npm run build` and embeds the static SPA. Backend devs are not blocked by frontend tooling.
- **Server-side sessions in SQLite**: opaque cookie ID, sliding expiry, immediate revocation by deleting the row. Touch-throttled (≥ 1 minute between updates) so authenticated requests don't pay a DB write each.
- **CSRF as synchronizer token bound to session**: two cookies (HttpOnly session ID, non-HttpOnly CSRF token); SPA echoes CSRF in `X-Csrf-Token` on unsafe methods; bearer-authed requests are exempt because cookies aren't auto-attached.
- **API keys stored as SHA-256 hash, raw shown once**: the only useful thing on disk after issuance is the hash; if the DB leaks, keys are not directly usable. Bearer scheme (`Authorization: Bearer tk_…`) for programmatic auth.
- **In-memory rate limiter for /login**: single-host product; distributed coordination would be over-engineering. State is reset on restart, which is acceptable for protecting against scripted brute force.
- **`docker compose` shell-out, not SDK reimplementation**: Compose has years of edge cases (depends_on, healthcheck merging, build context, profiles, secrets). Reproducing them in Go is wasted work; the docker-compose plugin is a non-negotiable host requirement and we already lean on Docker itself.
- **Traefik file-based dynamic config (NOT labels)**: with labels both blue + green stacks register routers and Traefik round-robins during the overlap, defeating the atomic flip. With files we explicitly own which colour serves traffic; the flip is a single atomic file rewrite.
- **Deployment lock = a deployment row in (pending, running)**: no separate lock table. Acquisition is a `BEGIN IMMEDIATE` + check-and-insert in a single transaction so concurrent requests can't both win.
- **Phase events in memory, not persisted**: the set of phases will grow (split health-check into pulling-image vs probe, add a "warming-up" sub-phase, etc.) and we don't want a schema migration each time. Coarse Status on the deployment row is enough for history; fine Phase serves the live UI only.
- **SSH + PAT in Phase 4, GitHub App deferred**: GitHub App auth needs a public callback URL, installation-token exchange, and a registered app at GitHub — realistic once Phase 7's installer establishes a DNS story. SSH deploy keys cover self-hosted CD with zero GitHub-side admin beyond pasting a key.
- **Uniform 401 on webhook failures**: unknown slug, missing webhook secret, and invalid signature all return the same status and body. An attacker enumerating slugs cannot distinguish the three cases.
- **Git-backed compose shadows stored compose**: when `git_url` is set, the engine reads from `<checkout>/<git_compose_path>`. The stored `compose_file` field remains editable (and is the source for git-less apps) but is not the source of truth when git is configured. One mental model: "the repo is the source of truth when you point me at a repo".
- **Public key derived on demand, not stored**: a private ed25519 PEM is sufficient to reconstruct the public half. Avoids a second column that could drift after rotation.
- **Shared env vars are opt-in per app, never universal**: a global "all apps see all shared keys" model is convenient but turns shared env into an audit-evading side channel. The `app_shared_env_vars` join table forces the question "which apps actually need this secret?" at write time.
- **Env values written to a file, not passed as `environment:`**: docker compose's `--env-file` keeps the values off the process command line, off `docker inspect` for ad-hoc container starts, and lets the user's compose `environment:` block stack on top without us having to merge YAML.
- **AAD constants live in `internal/domain`**: both the API (write path) and the engine (read path) call `domain.EnvVarAppAAD` / `domain.EnvVarSharedAAD`. Making the constants visible from one place prevents silent drift; putting them in a third "crypto-purposes" package would have leaked envvar semantics into a layer that doesn't care about them.
- **Traefik static config owned by Teal, restart required after ACME changes**: the static config is read at boot only; trying to hot-reload would mean shipping a sidecar or running Traefik in a mode we don't own. Generating the file from settings + surfacing a "restart Traefik" hint in the API response is the cheapest correct path.
- **HTTP-01 challenge only in v1**: HTTP-01 works against any DNS provider and needs no per-provider config. DNS-01 (with wildcards) deferred to Phase 7 once the installer covers DNS provisioning.
- **Volume delete requires typed-name confirmation**: misclicking a `Delete` button on a database volume should not be possible. Server-side check (`?confirm=<name>`) AND client-side typed prompt — defence in depth against both shoulder-surfing UI bugs and curl-then-regret API use.
- **One container watcher feeds everything**: container discovery is a single concern; metrics + log tailers consume the same Started/Stopped events. Each subsystem polling Docker independently would multiply load and create three subtly-different "what is a platform container" definitions.
- **Realtime hub is multiplexed over one WebSocket per tab**: spec calls for a "shared connection manager so multiple UI components can subscribe without opening multiple sockets". One socket + topic op messages keeps browser-side socket count predictable and makes subscribe/unsubscribe trivially cheap. Slow consumers drop events rather than block the producer; the next event implicitly tells them they missed something (and `_meta` carries the count).
- **Container logs persisted as per-container NDJSON files (not SQLite rows)**: log lines arrive at high velocity and the only access patterns are append, tail-from-offset, and prune-by-time. An NDJSON file with periodic linear-scan rewrite is dirt-simple and avoids one WAL flush per line. SQLite is used for metric samples (low rate, structured filtering) but would over-pay for raw log lines.
- **Container logs survive container removal for the retention window**: when a container disappears, its tailer stops but the file stays around for `IdleDelete` (defaults to retention). Lets users still grep recent logs from a crashed container — the most important time to be able to read them.
- **Historical replay is HTTP, live tail is WebSocket**: cleanest contract. UI fetches the last hour over HTTP, then opens the WS subscription for live lines. No special "give me history first" frame on the WS, no race between historical scan and the live cursor.
- **Engine's deploy.log is teed to the hub at the byte level**: every line written to `deploy.log` (including subprocess output captured from `docker compose build`) gets a `\n`-split publish on `deploy.<id>`. The persisted file remains the source of truth for cold reads; the hub is purely a fanout. Engine Phase events ride the same topic — one subscription per deploy gets both kinds of frames.
- **Outbound webhooks refuse to send unsigned**: with no per-app secret stored, the dispatcher logs and drops rather than POSTing an unauthenticated payload — receivers can't tell a real Teal event apart from a forgery, so we won't make one.
- **Resource limits applied to ALL services, not just the primary**: a sidecar can OOM the whole host; the spec says "per app", and treating that as one budget for the whole compose is the conservative read.
- **Marker-and-exit self-update**: in-process `docker pull && restart` from inside the container Teal runs in is brittle (pulls the image you're currently running, has to handle the gap when its own container is gone). Writing a marker + exiting + letting the supervisor reconcile is cheap and matches how every other long-running service handles "restart me".
- **Installer asks two questions; defaults the rest**: admin email and base domain are the only inputs that actually need a human. Port, data directory, version, log level all have sensible defaults; advanced flags exist (`--yes`, `--data-dir`, `--version`) but stay out of the happy path.
- **Notifications and audit_logs are separate concerns**: audit is forensic (immutable, every state change), notifications are user-facing (read state, dismissable, scoped). Conflating them would force per-user read tracking on every audit row and turn the audit query into a bell-feed query.
- **Metrics downsampling is raw→1m only in v1**: 6h raw + 18h aggregate buys the dashboards we have. Multi-tier (1m→1h) can come later when a real 7-day-history use case turns up.
